# Контекст Проекта: VR/AR Kvant Knowledge Hub

Ты — эксперт-разработчик (Frontend, Python, DevOps), работающий над образовательным порталом "VR/AR Kvant Knowledge Hub". Твоя задача — помогать развивать проект, строго следуя описанной архитектуре и правилам.

## 🛠 Технологический стек (СТРОГИЕ ОГРАНИЧЕНИЯ)
- **Frontend:** ТОЛЬКО чистый Vanilla JS (ES6 модули), HTML5 и CSS3. 
- **Стилизация:** Tailwind CSS (подключен через CDN в `index.html`). Не используй npm-пакеты для Tailwind.
- **Контент:** Markdown-файлы с YAML Frontmatter (метаданные).
- **Сборка данных:** Python (`scripts/build_manifests.py`) используется для генерации JSON-манифестов (`tracks.json`, `cheats.json`, `portfolio.json`).
- **CMS:** Decap CMS (Static CMS) для управления контентом без бэкенда.
- **ЗАПРЕЩЕНО:** Использовать React, Vue, Angular, сборщики (Webpack/Vite) или добавлять тяжелые npm-зависимости без явного разрешения. Проект должен оставаться максимально легковесным SPA.

## 📂 Архитектура директорий
- `/` - Корневая директория (здесь лежат `index.html`, `ROADMAP.md`, `GEMINI.md`).
- `/js/` - ES6 модули фронтенда (`main.js`, `router.js`, `markdown.js`, `ui.js`, `api.js`).
- `/css/` - Базовые стили (`style.css`).
- `/articles/` - База знаний. Внутри лежат `.md` файлы по папкам треков (`blender/`, `unity/`, `unreal/`) и JSON-манифесты.
- `/admin/` - Панель управления Decap CMS (`index.html`, `config.yml`).
- `/scripts/` - Скрипты автоматизации (например, `build_manifests.py`).
- `/.github/workflows/` - CI/CD пайплайны.

## ⚠️ Критические правила работы с кодом
1. **Кастомный Markdown:** Во фронтенде (в `js/markdown.js`) есть уникальные регулярные выражения для парсинга тегов `[gallery: ...]` и `[compare: ...]`. НИКОГДА не удаляй и не ломай эту логику при рефакторинге!
2. **Синхронизация данных:** База данных сайта строится на файлах `.json` в папке `/articles/`. Эти файлы генерируются АВТОМАТИЧЕСКИ. Если ты меняешь структуру `.md` файлов или добавляешь новые статьи в рамках задачи, ты обязан запустить (или обновить) скрипт `scripts/build_manifests.py`.
3. **Глобальный Scope:** При экспорте функций из ES6 модулей, которые вызываются напрямую из HTML (через `onclick`), обязательно прокидывай их в `window` (например, `window.goBackSafe = goBackSafe`).

## 📝 Специфические правила Decap CMS (`config.yml`)
1. **Генерация путей (Slugs):** НИКОГДА не используй `slug: "{{title}}"`, так как проект на русском языке, и это создаст файлы с кириллицей и пробелами. ВСЕГДА используй `slug: "{{slug}}"` (CMS сама транслитерирует заголовок).
2. **Фильтрация (Filters):** Decap CMS НЕ поддерживает сложные регулярные выражения (regex) с Lookahead/Lookbehind в поле `filter.match`. Для разделения сущностей (например, обычные уроки и `intro.md`) используй проверку точного значения скрытого поля (например, `type: "lesson"`).
3. **Бэкенд (Backend):** По умолчанию для проектов без собственного сервера авторизации используй `name: git-gateway` (для хостинга на Netlify).

## 🔄 Пайплайн выполнения задач (STRICT GIT FLOW)

Твоя главная задача — автономно выполнять пункты из файла `ROADMAP.md`. При выполнении любой задачи ты ОБЯЗАН строго соблюдать следующий алгоритм:

**Шаг 1: Подготовка и Ветвление**
- Перед началом проверь, что ты находишься в ветке `main` и она обновлена (`git pull origin main`).
- Создай новую ветку для задачи. Используй понятные названия с префиксами: `feature/` (новый функционал), `fix/` (исправление ошибок), `content/` (статьи), `config/` (настройки).
  *Пример: `git checkout -b feature/setup-decap-cms`*

**Шаг 2: Выполнение задачи и Коммиты**
- Напиши код, создай или измени нужные файлы согласно задаче из ROADMAP.md.
- Сделай один или несколько аккуратных коммитов в текущую ветку. Сообщения коммитов пиши на русском языке (например: `feat: добавлен config.yml для CMS`).

**Шаг 3: ОСТАНОВКА И ЗАПРОС РЕВЬЮ (КРИТИЧЕСКИ ВАЖНО)**
- Как только работа в ветке закончена и закоммичена, ТЫ ДОЛЖЕН ОСТАНОВИТЬСЯ.
- Выведи пользователю в терминал сообщение: *"Задача выполнена в ветке [имя_ветки]. Пожалуйста, проверь изменения в редакторе. Напиши мне 'ок', 'сливай' или 'merge', если всё работает правильно. Если есть ошибки, укажи, что нужно исправить."*
- **ЗАПРЕЩЕНО** сливать ветку в `main` без текстового подтверждения от пользователя!

**Шаг 4: Слияние (Merge) и Синхронизация**
- Увидев команду разрешения от пользователя (например, "ок" или "merge"), переключись на ветку `main` (`git checkout main`).
- Слей изменения из рабочей ветки (`git merge [имя_ветки]`).
- Отправь изменения в удаленный репозиторий (`git push origin main`).
- Удали локальную рабочую ветку (`git branch -d [имя_ветки]`).

**Шаг 5: Обновление статуса проекта**
- Открой файл `ROADMAP.md`.
- Поставь галочку `[x]` напротив задачи, которую ты только что успешно завершил.
- Сделай финальный коммит напрямую в main: `docs: обновлен ROADMAP (выполнен Шаг X)`.
- Сделай пуш (`git push origin main`).
